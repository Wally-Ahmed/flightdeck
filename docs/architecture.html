1 The problem

Turning an idea into shipped software is slow, manual, and easy to get wrong

When someone asks for a new feature, a lot has to happen by hand. Someone reads the request and figures out a plan. Someone writes the code. Someone tests it. Someone sets up a place to try it. Someone double-checks each step. Every handoff is a chance for things to stall — or for a mistake to slip through unnoticed until much later.

The usual way
  • 🐌A person reads the request and writes a plan — when they have time.
  • ⌨️A developer writes the code by hand, often days later.
  • 🔁Testing is a separate manual step that's easy to skip under deadline.
  • 🚧Setting up a place to try the change live takes its own effort.
  • 🕳️A broken step can slip through and only surface much later.
→
The Flightdeck way
  • ⚡AI workers plan, code, and test automatically, in minutes.
  • ✅A quality check runs after every step — automatically, every time.
  • 🌐A live preview is built and deployed for you to click and try.
  • 🛑If any step fails, the line stops right there — no bad work ships.
  • 👤A human gets a tested change with a link to try it before approving.
2 The big idea

Think of it as an automated assembly line for software

A car factory takes raw parts in one end and rolls a finished, inspected car out the other — each station does one job, and an inspector checks the work before it moves down the line. Flightdeck does the same thing with software: an idea enters one end, passes through stations that plan it, build it, test it, and ship a preview, and a quality inspector signs off after every single station. A finished, tested feature rolls out the other end.

Sponsor · hosting + live previews

Render is where every result turns into something a human can click and try

The factory's promise is "a link where someone can see and test it." That link is Render's job. For each issue the factory implements, Render stands up a live, clickable preview environment and the pull request links straight to it. Render also runs the factory itself — the orchestrator, the run board, and the database all live here.

The starring role

Every issue becomes a live, clickable, testable preview — linked on the pull request

When the code is built and proven, Render takes that exact version and puts it on the internet as a temporary live copy with its own web address. The factory comments that address on the pull request. One click and a reviewer is using the real feature — no checkout, no setup, nothing to install. That is, literally, the project's deliverable.

The previews are spun up on demand via the Render API as each run reaches its deploy stage — not pre-declared, but created at runtime, one per issue.

+ And it hosts the whole factory

Three more things run on Render, all the time

Beyond the per-issue previews, Render is the home of the factory's permanent pieces — defined as code in render.yaml.

The orchestrator

web service · node

The foreman — ingests the issue, hosts the coding agent, calls the Render API, opens the PR, serves the run-board API.

The run board

static site

The dashboard a presenter points at — a React/Vite site, served by Render, reading live run state from the orchestrator.

The database

postgres · flightdeck-db

Durable run-state storage — mirrors the Superplane runs memory so the board keeps history across restarts.

The permanent services are declared in render.yaml; the per-issue previews are intentionally not — they're created at runtime by the orchestrator through the Render API, one per run.

What's a "pull request"?

It's just the standard, polite way a developer proposes a change: "here's what I changed, here's why — take a look before it goes in." Nothing happens to the real project until a human approves it. Flightdeck opens one for you, fully filled in.

What's a "preview environment"?

A temporary, live copy of the feature you can click and try — its own web address on Render, separate from anything real. Like a test-drive car on the lot: fully functional, but safe to poke at. When you're done, it can be thrown away.
