1 The problem

Turning an idea into shipped software is slow, manual, and easy to get wrong

When someone asks for a new feature, a lot has to happen by hand. Someone reads the request and figures out a plan. Someone writes the code. Someone tests it. Someone sets up a place to try it. Someone double-checks each step. Every handoff is a chance for things to stall — or for a mistake to slip through unnoticed until much later.

The usual way
  • 🐌A person reads the request and writes a plan — when they have time.
  • ⌨️A developer writes the code by hand, often days later.
  • 🔁Testing is a separate manual step that's easy to skip under deadline.
  • 🚧Setting up a place to try the change live takes its own effort.
  • 🕳️A broken step can slip through and only surface much later.
→
The Flightdeck way
  • ⚡AI workers plan, code, and test automatically, in minutes.
  • ✅A quality check runs after every step — automatically, every time.
  • 🌐A live preview is built and deployed for you to click and try.
  • 🛑If any step fails, the line stops right there — no bad work ships.
  • 👤A human gets a tested change with a link to try it before approving.
2 The big idea

Think of it as an automated assembly line for software

A car factory takes raw parts in one end and rolls a finished, inspected car out the other — each station does one job, and an inspector checks the work before it moves down the line. Flightdeck does the same thing with software: an idea enters one end, passes through stations that plan it, build it, test it, and ship a preview, and a quality inspector signs off after every single station. A finished, tested feature rolls out the other end.

The diamond ◆ after every station is a quality checkpoint. It lights up green only when the previous station's work is proven good. That's the whole secret: nothing advances on faith.

The workers are AI

The planning, coding, and testing are done by AI agents — software that can read a request and do the actual work, the way a skilled person would, but in minutes.

Checks are the point

The headline feature isn't the AI — it's the checkpoint after every step. A broken plan never reaches coding; broken code never reaches testing; a dead preview never reaches a human.

You get something real

The output isn't a description — it's a working preview you can actually click and try, plus a tidy proposed change ready for a person to glance over and approve.

4 The two engines

Two partners make it run — both are essential

The factory needs two things to work: someone to run the line and enforce every inspection, and somewhere to put the live preview so a human can click it. Those are the two sponsor technologies — and the project genuinely can't work without either one.

Superplane

The conductor — runs the line & enforces every check
Sponsor · orchestration

Superplane is the control tower. It runs the stations in the right order, hands the work from one to the next, and — most importantly — enforces the quality checkpoint after every step. If a check fails, Superplane stops that request cold. It keeps one independent run going for each request, all at the same time, and it remembers exactly where each one is.

🎼Conducts the steps. Runs plan → code → test → preview → propose, in order, one run per request.
🛡️Enforces every check. After each step it asks "did that actually work?" — and only then moves on.
🧠Does some thinking itself. It writes the plan and the review notes using a built-in AI step.
⚙️Technically: a 15-node canvas.yaml — native claude.textPrompt steps for plan & review, if / health-check gates after every stage, and a built-in console run board reading the runs memory.

Render

The showroom — hosts the live preview you click
Sponsor · hosting + previews

Render is the showroom floor. When the code is built and tested, Render takes that exact version and puts it on the internet as a temporary live copy — its own web address. That's the "preview environment": a real, running version of the feature you can open in a browser and click around, without touching anything real. Render also hosts the factory's own control panel.

🌐Puts the preview online. Each request gets its own live web address to open and try.
⚡Instant & safe. A throwaway copy — clicking around it can't break anything real.
🏠Houses the whole factory. The control panel and behind-the-scenes machinery run here too.
⚙️Technically: component features ship as a static Storybook site; richer flows ship as a Docker-image web service via POST /v1/services → serviceDetails.url. A live health-check gate confirms it's actually up.
Superplane runs the line & checks every step + Render serves the preview a human clicks = an idea becomes a working, reviewable feature — on its own.

The foreman (the orchestrator)

Between the two engines sits a small piece of custom software — think of it as the shop foreman. When the conductor says "now write the code" or "now deploy the preview", the foreman is who actually rolls up its sleeves and does that heavy work, then reports back whether it succeeded.

Technically: a TypeScript service exposing /stages/code · /verify · /deploy · /pr that the canvas's http nodes call; it hosts the coding agent, talks to GitHub and the Render API, and serves the board's /api/runs.

The coding worker (the AI agent)

The star employee is an AI coding agent: it takes the plan, opens the project, writes the actual code changes, and checks its own work builds — just like a developer would, but automatically. This is the hardest, most impressive part of the whole line.

Technically: the Claude Agent SDK running headless — clone the fork, edit web_src/, build, commit, push a branch, return CodeResult{ buildPassed, summary, headSha }. A canned #5368 change is kept as insurance so the line always demos.

What's a "pull request"?

It's just the standard, polite way a developer proposes a change: "here's what I changed, here's why — take a look before it goes in." Nothing happens to the real project until a human approves it. Flightdeck opens one for you, fully filled in.

What's a "preview environment"?

A temporary, live copy of the feature you can click and try — its own web address, separate from anything real. Like a test-drive car on the lot: fully functional, but safe to poke at. When you're done, it can be thrown away.
